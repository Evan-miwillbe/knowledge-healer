# Knowledge Healer

**知识库自愈巡检系统** — 一个 Claude Code Skill，为飞书知识库提供多Agent并行健康诊断

新人入职第一天，主管说"去看wiki上的报销流程"。照着走了一遍，被财务打回来。主管说"哦那个早改了，直接提交就行"。

> "过时文档消耗的时间比没有文档还多，因为人们会按错误步骤操作、搞坏东西，然后需要老员工花时间来修复。"
> — r/ExperiencedDevs, 2025

这不是个案。调研显示 **70%以上企业知识库"几乎无人用或仅勉强有人用"**。根本原因很简单：写一篇文档是一次性成本，人们愿意付；维护一篇文档是持续成本，没人愿意管。

Knowledge Healer做一件事：**让机器替你检查，你只管修。**

## 解决什么问题

以下每个痛点都来自真实用户反馈（Reddit、知乎、V2EX、人人都是产品经理，2021-2025）：

### 过期误导 — "陈旧文档比没有文档更危险"

> "新人花了两天按照安装文档操作，结果文档引用的工具2023年就废弃了。"（r/ExperiencedDevs）
>
> "架构文档不更新、接口说明落后于实际实现、README只停留在初始化版本。"（知乎）

**Knowledge Healer → 过期检测**：不是"90天没编辑就标红"——会议纪要三年不更新也正常，入职指南30天没改可能就出问题了。按文档类型区分有效期，多信号加权判定。

### 版本混乱 — "搜出三个版本，不知哪个是最新的"

> "运营部同事搜索'裂变'能搜出来三个版本的复盘文档，根本不知道哪个是最新的。"（知乎）
>
> "上百个Space不知道内容该放哪，同一话题有多个重复页面在不同位置。"（r/softwaredevelopment）

**Knowledge Healer → 冲突检测**：对同主题文档做关键断言对比，自动识别哪个是权威源。

### 群聊黑洞 — "我们的知识库其实是Slack搜索"

> "有Confluence，有专人负责过一个季度，有模板和空间分类体系。没人用。需要装VPN？搜Slack。需要软件许可流程？搜Slack。"（r/sysadmin）
>
> "技术部同事花两个周末整理了详细API文档发到群里，没人看，过了一个月他自己都忘了。"（知乎）

**Knowledge Healer → 管线泄漏检测**：扫描群聊中被反复回答的高价值内容，对照wiki看有没有收录。同一问题被问3次以上且wiki无对应页面 = P0泄漏。

### 文档漂移 — "文档没变，但我们早就不这么做了"

> "每次发布功能或重构后文档就落后了。我们都知道应该更新，但总有更紧急的事。三个月后新人加入，花两天和过时的安装指南搏斗。"（r/devops）
>
> "新人入职第一周，不是在熟悉代码，而是在四处找人问'账号怎么申请'、'测试环境怎么连'。"（知乎）

**Knowledge Healer → 漂移检测**（核心创新）：去群聊里搜索反驳信号——有人说"我们现在不这么做了"但文档还写着旧流程。跨源交叉验证，这是纯文档检查工具做不到的。

### 入口堵塞 — "收集了资料但扔在那里几个月不动"

> "员工心声：'工作已经很饱和了，还要我花额外时间写总结文档，太麻烦了。'"（EKB调研）
>
> "季度文档冲刺——大家参与3天然后停了。关闭工单时必须更新文档——执行率约20%。"（r/sysadmin）

**Knowledge Healer → 捕获箱老化检测**：扫描待处理区，计算入口流量 vs 处理速率。堆得越多越不想处理，系统在恶性循环开始前就发出警告。

### 框架陷阱 — "目录漂亮，里面全是空的"

> "知识库试点项目失败的真正原因不是技术不行，而是上下文切换成本太高——用户正在群里答疑，却要切换到浏览器打开知识库。"（CSDN）
>
> "15人团队，Confluence 400+页面，60%过时。新人被告知'wiki有一切'，然后困惑一周。"（r/Notion）

**Knowledge Healer → 覆盖空洞检测**：识别有标题无内容的骨架页面，结合群聊需求信号判断优先级——被反复问到的空洞比纯结构占位的空洞紧急得多。

### 额外发现：离职知识断层

> "核心架构团队集体离职，系统维护停滞，新功能开发延迟率从15%暴涨到65%。"（CSDN）
>
> "我问的每个问题答案都是'问某某某'——而某某某已经不在了。"（r/ExperiencedDevs）

**Knowledge Healer → 管线泄漏 + 知识复利指数**：泄漏检测发现只存在于个人脑中/群聊中的关键知识，在人员离职前就推动录入wiki。知识复利指数量化"知识是在积累还是只在个人身上"。

## 一个新场景：AI知识库的前置清洗

> "搭建AI知识库踩了37个坑：原始文档自身质量问题——逻辑混乱、版本并行、术语不统一。检索准确率仅65%。"（53AI, 2025）

越来越多团队在搭建RAG/AI知识库——但**垃圾进垃圾出**。如果源文档本身过期、矛盾、散落，AI只会更高效地给出错误答案。Knowledge Healer可以作为AI知识库的前置质量关卡：先治好文档病，再喂给AI。

## 核心能力

### 七维健康诊断

| 维度 | 检测什么 | 对应真实痛点 | 创新点 |
|------|---------|------------|-------|
| **过期** | 文档不再反映当前状态 | 过期误导 | 按文档类型区分有效期，多信号加权 |
| **冲突** | 多份文档描述不一致 | 版本混乱 | 主题聚类后自动识别权威源 |
| **断链** | 链接指向已删除的目标 | 找不到 | 含隐式引用模糊匹配 |
| **漂移** | 文档没变但现实已变 | 文档漂移 | **跨源交叉验证**：文档 vs 群聊 |
| **管线泄漏** | 高价值知识散落在体系外 | 群聊黑洞 | 群聊高价值内容扫描+wiki覆盖度匹配 |
| **捕获箱老化** | 入口堵塞 | 入口堵塞 | 入口流量 vs 处理速率 |
| **覆盖空洞** | 有框架无内容 | 框架陷阱 | 结合群聊需求信号判断填充优先级 |

管线泄漏发现"体系外散落的内容"，覆盖空洞发现"体系内缺失的位置"——两者自动匹配，生成"从哪取内容填到哪"的修复方案。

### 3+2 多Agent并行架构

3个必选Agent + 2个条件激活Agent：

```
主CC（调度者 + 综合者）
 ├─ Agent-Structure     过期 + 断链（共享元数据上下文，合并执行）
 ├─ Agent-Contradiction 冲突检测（文档≥10篇时激活）
 ├─ Agent-Drift         漂移检测（配置chat_groups时激活）
 ├─ Agent-Leakage       泄漏检测（配置chat_groups时激活）
 └─ 主CC直接执行         捕获箱老化 + 覆盖空洞
```

**架构设计要点**：

- **条件激活**：预扫描元数据后决定启动哪些Agent，跳过无数据源的检测维度
- **Context隔离**：每个Agent只接收本维度所需数据，不传入全量文档
- **三层递进扫描**：元数据层(5% token) → 内容层(35%) → 跨源层(60%)
- **增量巡检**：有基线时只检查变化部分，续检token减少50-80%
- **Prompt Caching对齐**：稳定前缀 + 可变后缀，跨次巡检命中缓存

### 自校准机制

系统从反馈中学习，动态调整判定阈值：

- 用户标记"误报" → 放宽阈值
- 发现"漏报" → 收紧阈值
- 连续3次同方向调整 → 锁定为稳定阈值
- **噪声控制**：宁可漏报不可误报——报告50%是假警报，用户很快就不看了

### 知识复利指数

量化知识库的新陈代谢速率：

| 指标 | 衡量什么 |
|------|---------|
| 录入率 | 新内容有多少真正进入了正式体系 |
| 引用率 | wiki里的东西有人在用吗 |
| 更新率 | 知识在迭代还是写完就没人管了 |
| 泄漏率 | 有多少知识产生了但绕过了体系 |

综合指数 > 0.7 健康增长，0.4-0.7 需要关注，< 0.2 名存实亡。

## 技术差异化

| | 人工审查 | 文档检查工具 | Knowledge Healer |
|---|---------|-----------|-----------------|
| 触发方式 | 人工启动 | 规则触发 | 定时自动 + 按需 |
| 过期判定 | 主观经验 | 固定天数阈值 | 按文档类型 × 多信号加权 |
| 数据源 | 只看文档 | 只看文档 | 文档 + 群聊 + 任务，跨源交叉验证 |
| 检测范围 | 已有文档有没有问题 | 已有文档有没有问题 | 同时发现"该有但没有的内容" |
| 判定标准 | 固定 | 固定 | 自校准，越用越准 |
| 输出 | 问题清单 | 问题清单 | 问题 + 证据 + 修复建议 + 趋势量化 |
| 学习能力 | 无 | 无 | 从误报/漏报反馈中动态调整 |

## 使用方法

### 前置条件

- [飞书 CLI](https://github.com/AJ-Lark/feishu-cli)（需要 Wiki/Doc/IM 权限）
- [Claude Code](https://docs.anthropic.com/en/docs/claude-code)

### 安装

```bash
git clone https://github.com/Evan-miwillbe/knowledge-healer.git ~/.claude/skills/knowledge-healer
```

### 带漂移检测的完整用法

```
/knowledge-healer
scope: wiki://your-space-token
project_dir: ~/Desktop/knowledge-audit
mode: full
depth: 3
chat_groups: ["oc_product_team", "oc_engineering"]
notify: true
baseline_path: ~/Desktop/knowledge-audit/baselines/latest.json
```

### 执行模式

| 模式 | 适用场景 | 耗时（50篇文档） |
|------|---------|----------------|
| `full` | 首次运行 / 季度体检 | ~5分钟 |
| `quick` | 每周例行巡检 | ~1分钟 |
| `drift-only` | 专项检测知识漂移 | ~3分钟 |
| `scheduled` | 无人值守定期巡检 | 同full，静默推送 |

## 适用场景

| 场景 | 对应痛点 | 怎么用 |
|------|---------|-------|
| 新人入职保障 | 入职文档过期/漂移 | 对入职指南空间做巡检，确保内容最新 |
| 季度知识盘点 | wiki变垃圾场 | `mode: full` 全面体检 |
| 人员离职前 | 知识断层风险 | 泄漏检测发现只在个人身上的关键知识 |
| AI知识库搭建前 | 源文档质量差 | 先治好文档病，再喂给RAG |
| 合规审计准备 | 文档与实际操作不一致 | 漂移检测确保文档反映真实流程 |
| 跨部门协作 | 多版本矛盾 | 发现不同部门wiki间的冲突 |
