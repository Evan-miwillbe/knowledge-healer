# Knowledge Healer

新人入职第一天，主管说"去看wiki上的报销流程"。新人照着走了一遍，被财务打回来。新人去问主管，主管说"哦那个早改了，直接提交就行"。

"那wiki上为什么还写着旧流程？"

"谁有空改那个。"

这个场景每天都在发生。不是个案，是必然。

**wiki在你写完的那一刻就开始死了。** 不是因为没人写——写一篇文档是一次性成本，人们愿意付。但维护一篇文档是持续成本：每隔三十天检查一遍、和实际操作对一遍、把过时的部分改掉。人性的弱点就在这里：一次性成本能忍，持续成本本能回避。

Knowledge Healer做一件事：**让机器替你检查，你只管修。**

## 跑一次你会看到什么

```
/knowledge-healer
scope: wiki://your-space-token
project_dir: ~/Desktop/knowledge-audit
mode: full
depth: 3
```

五分钟后，你拿到一份体检报告：

> **知识库健康报告 2026-05-05**
>
> 健康评分：**72/100** (↑5 vs上次)
> 知识复利指数：**0.45** 🟡 增长放缓
> 文档总数：156 篇

| 类别 | 总数 | P0 | P1 | P2 |
|------|------|----|----|-----|
| 过期 | 12 | 2 | 5 | 5 |
| 冲突 | 3 | 1 | 2 | 0 |
| 断链 | 8 | 0 | 3 | 5 |
| 漂移 | 4 | 2 | 2 | 0 |
| 管线泄漏 | 6 | 2 | 3 | 1 |
| 捕获箱老化 | 5 | 0 | 2 | 3 |
| 覆盖空洞 | 4 | 1 | 2 | 1 |

P0是必须修的，P1是该修的，P2是留意就行。每条发现附带证据和修复建议：

> **P0-漂移**: 报销流程文档与实际操作不一致
> - 证据: 3人在财务群提及"直接提交"流程
> - 影响: 新员工按旧流程操作被退回
> - 建议: @张三 更新报销SOP第3步
>
> **P0-泄漏**: "如何配置VPN"在技术群被问了7次
> - 证据: 5人分别回答，内容高度一致但wiki无此页面
> - 影响: 每次新人入职都重复问+重复答
> - 建议: 整理群聊回答，创建wiki页面（预估15分钟）

下个月再跑一次，问题从27个降到8个。这就是自愈。

## 它检查什么

七件事，对应知识库的七种死法：

**过期** — 文档还在，但内容已经不对了。不是"90天没编辑就算过期"那种蠢规则——会议纪要三年不更新也正常，但入职指南30天没改可能就出问题了。系统会区分文档类型，按不同节奏判断。

**冲突** — 两份文档对同一件事说法不一样。到底听谁的？系统找出来，告诉你哪个是最新版。

**断链** — 文档里的链接点进去是404，或者已经搬家了。

**漂移** — 这是最有价值的一项。系统去群聊里搜，看有没有人说"我们现在不这么做了"，但文档还写着旧流程。文档没变，现实变了——这种偏差靠人工审查根本发现不了。

**管线泄漏** — 群聊里有人反复回答同一个问题，说明这个知识应该写进wiki但没人写。知识在产生，但绕过了体系，集体智慧没有沉淀。

**捕获箱老化** — 有人收集了资料，扔在"待处理"文件夹里两个月不动。入口堵了，新知识永远进不了正式体系。堆得越多越不想处理，恶性循环。

**覆盖空洞** — wiki目录看着很完整，点进去全是空页面。框架漂亮，里面是空的。比没有目录更糟——因为它给人"这块有人在管"的错觉。

泄漏发现的是"体系外散落的内容"，空洞发现的是"体系内缺失的位置"。两者匹配起来，就知道该从哪里取内容填到哪里。

## 它怎么做到的

派出一队AI分头行动，各管各的：

- **结构健康Agent** — 查过期和断链（这两项共享文档元数据，合在一起效率高）
- **冲突检测Agent** — 对比同主题文档的关键断言（文档太少时自动跳过，没有统计意义）
- **漂移检测Agent** — 去群聊里找反驳证据（没配群聊ID就不启动，不做无米之炊）
- **泄漏检测Agent** — 扫描群聊中的高价值内容，对照wiki看有没有收录

捕获箱老化和覆盖空洞由主程序直接处理，不需要单独派Agent——纯粹看元数据就够了。

每个Agent只拿到自己需要的数据，互不干扰。做完把结论（不超过200字）交回来，主程序交叉验证后生成最终报告。

不是所有巡检都需要全部Agent——系统先做一轮低成本的预扫描，决定这次该启动谁。文档少于10篇？跳过冲突检测。没配群聊？跳过漂移和泄漏。跑过一次有了基线？只检查变化部分。

**越用越准。** 你说"这个不算过期"，下次它自动放宽标准。你发现它漏了一个问题，下次它收紧标准。连续三次同方向调整后锁定，不再频繁变动。宁可漏报也不误报——如果报告一半是假警报，你很快就不看了。

## 安装

需要先装好 [飞书 CLI](https://github.com/AJ-Lark/feishu-cli) 和 [Claude Code](https://docs.anthropic.com/en/docs/claude-code)，并配置飞书授权（Wiki、Doc、IM权限）。

```bash
git clone https://github.com/Evan-miwillbe/knowledge-healer.git ~/.claude/skills/knowledge-healer
```

## 执行模式

| 模式 | 什么时候用 | 50篇文档大概要多久 |
|------|---------|----------------|
| `full` | 第一次跑，或者季度体检 | 5分钟 |
| `quick` | 每周例行巡检 | 1分钟 |
| `drift-only` | 专门查"文档和实际操作对不对得上" | 3分钟 |
| `scheduled` | 设一次，以后自动跑，结果推到群里 | 同full |

带漂移检测的完整用法：

```
/knowledge-healer
scope: wiki://your-space-token
project_dir: ~/Desktop/knowledge-audit
mode: full
depth: 3
chat_groups: ["oc_product_team", "oc_engineering"]
notify: true
baseline_path: ~/Desktop/knowledge-audit/baselines/latest.json
```

## 知识复利模型

你的知识库是在成长还是在萎缩？四个指标算出一个数：

- **录入率** — 新内容有多少真正进入了正式体系
- **引用率** — wiki里的东西有人在用吗
- **更新率** — 知识在迭代还是写完就没人管了
- **泄漏率** — 有多少知识产生了但绕过了体系

指数大于0.7是健康的，0.4-0.7需要关注，低于0.2基本名存实亡。
