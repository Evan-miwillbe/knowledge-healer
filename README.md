# Knowledge Healer

**知识库自愈巡检系统** — 一个 Claude Code Skill，为飞书知识库提供多Agent并行健康诊断

新人入职第一天，主管说"去看wiki上的报销流程"。照着走了一遍，被财务打回来。主管说"哦那个早改了，直接提交就行"。

这个场景每天都在发生。wiki在你写完的那一刻就开始死了——写一篇文档是一次性成本，人们愿意付；维护一篇文档是持续成本，没人愿意管。

Knowledge Healer做一件事：**让机器替你检查，你只管修。**

## 核心能力

### 七维健康诊断

不只查过期——同时检测知识库的七种病症：

| 维度 | 检测什么 | 创新点 |
|------|---------|-------|
| **过期** | 文档内容不再反映当前状态 | 按文档类型区分有效期，多信号加权判定 |
| **冲突** | 多份文档对同一事物描述不一致 | 主题聚类后自动识别权威源 |
| **断链** | 链接指向已删除或已移动的目标 | 含隐式引用模糊匹配 |
| **漂移** | 文档没变但现实已变 | **跨源交叉验证**：文档断言 vs 群聊讨论 |
| **管线泄漏** | 高价值知识散落在体系外 | 群聊高价值内容扫描，匹配wiki覆盖度 |
| **捕获箱老化** | 入口堵塞，新信息进不了体系 | 入口流量 vs 处理速率的健康度计算 |
| **覆盖空洞** | 有框架无内容的空壳区域 | 结合群聊需求信号判断优先级 |

管线泄漏发现"体系外散落的内容"，覆盖空洞发现"体系内缺失的位置"——两者自动匹配，生成"从哪取内容填到哪"的修复方案。

### 3+2 多Agent并行架构

3个必选Agent + 2个条件激活Agent，按诊断维度分工：

```
主CC（调度者 + 综合者）
 ├─ Agent-Structure     过期 + 断链（共享元数据上下文，合并执行）
 ├─ Agent-Contradiction 冲突检测（文档≥10篇时激活）
 ├─ Agent-Drift         漂移检测（配置chat_groups时激活）
 ├─ Agent-Leakage       泄漏检测（配置chat_groups时激活）
 └─ 主CC直接执行         捕获箱老化 + 覆盖空洞（纯元数据分析）
```

**架构设计要点**：

- **条件激活**：预扫描元数据后决定启动哪些Agent，跳过无数据源的检测维度
- **Context隔离**：每个Agent只接收本维度所需数据，不传入全量文档
- **三层递进扫描**：元数据层(5% token) → 内容层(35%) → 跨源层(60%)，大部分健康文档在第一层排除
- **增量巡检**：有基线时只检查变化部分，续检场景token减少50-80%
- **Prompt Caching对齐**：稳定前缀（角色+标准）+ 可变后缀（本次数据），跨次巡检命中缓存

### 自校准机制

系统从每次巡检的反馈中学习，动态调整判定阈值：

- 用户标记"误报" → 放宽该类型阈值
- 发现"漏报" → 收紧阈值
- 连续3次同方向调整 → 锁定为稳定阈值
- 单次调整幅度不超过20%，防止过拟合
- **噪声控制原则**：宁可漏报不可误报——报告50%是假警报，用户很快就不看了

### 知识复利指数

量化知识库的新陈代谢速率：

| 指标 | 衡量什么 |
|------|---------|
| 录入率 | 新内容有多少真正进入了正式体系 |
| 引用率 | wiki里的东西有人在用吗 |
| 更新率 | 知识在迭代还是写完就没人管了 |
| 泄漏率 | 有多少知识产生了但绕过了体系 |

综合指数 > 0.7 健康增长，0.4-0.7 需要关注，< 0.2 名存实亡。

| 覆盖空洞 | 4 | 1 | 2 | 1 |

每条P0/P1发现附带证据和修复建议：

> **P0-漂移**: 报销流程文档与实际操作不一致

>
> **P0-泄漏**: "如何配置VPN"在技术群被问了7次
> - 证据: 5人分别回答，内容高度一致但wiki无此页面
> - 影响: 每次新人入职都重复问+重复答
> - 建议: 整理群聊回答，创建wiki页面（预估15分钟）

## 技术差异化

| | 传统方案（人工审查） | 文档检查工具 | Knowledge Healer |
|---|---------|-----------|-----------------|
| 触发方式 | 人工启动 | 规则触发 | 定时自动 + 按需 |
| 过期判定 | 主观经验 | 固定天数阈值 | 按文档类型 × 多信号加权 |
| 数据源 | 只看文档本身 | 只看文档本身 | 文档 + 群聊 + 任务，跨源交叉验证 |
| 检测范围 | 已有文档有没有问题 | 已有文档有没有问题 | 同时发现"该有但没有的内容" |
| 判定标准 | 固定 | 固定 | 自校准，越用越准 |
| 输出 | 问题清单 | 问题清单 | 问题 + 证据 + 修复建议 + 趋势量化 |
| 学习能力 | 无 | 无 | 从误报/漏报反馈中动态调整 |

**核心创新**：漂移检测。过期、冲突、断链都能从文档元数据推断——漂移需要跨数据源交叉验证，对比"文档写的"与"群聊里大家实际做的"，这是纯文档检查工具做不到的。

## 使用方法

### 前置条件

- [飞书 CLI](https://github.com/AJ-Lark/feishu-cli)（需要 Wiki/Doc/IM 权限）
- [Claude Code](https://docs.anthropic.com/en/docs/claude-code)

### 安装

```bash
git clone https://github.com/Evan-miwillbe/knowledge-healer.git ~/.claude/skills/knowledge-healer
```

### 基本用法

```
/knowledge-healer
scope: wiki://your-space-token
project_dir: ~/Desktop/knowledge-audit
mode: full
depth: 3
```

### 带漂移检测的完整用法

```
/knowledge-healer
scope: wiki://your-space-token
project_dir: ~/Desktop/knowledge-audit
mode: full
depth: 3
chat_groups: ["oc_product_team", "oc_engineering"]
notify: true
baseline_path: ~/Desktop/knowledge-audit/baselines/latest.json
```

### 执行模式

| 模式 | 适用场景 | 耗时（50篇文档） |
|------|---------|----------------|
| `full` | 首次运行 / 季度体检 | ~5分钟 |
| `quick` | 每周例行巡检 | ~1分钟 |
| `drift-only` | 专项检测知识漂移 | ~3分钟 |
| `scheduled` | 无人值守定期巡检 | 同full，静默推送 |

## 适用场景

| 场景 | 怎么用 |
|------|-------|
| 季度知识盘点 | `mode: full` 全面体检 |
| 新人入职保障 | 对入职指南空间做巡检，确保内容最新 |
| 项目收尾 | `mode: drift-only` 检测文档是否反映最终状态 |
| 合规审计准备 | 先自查，确保文档与实际操作一致 |
| 跨部门协作 | 发现不同部门wiki间的信息冲突 |
| 知识复利启动 | 找出知识库"不长"的根因——入口堵？管线漏？框架空？ |

## 设计灵感

- [Karpathy's LLM Wiki](https://gist.github.com/karpathy/442a6bf555914893e9891c11519de94f) — "Obsidian是IDE，LLM是程序员，Wiki是代码库"
- 知识复利理论 — 每次交互让知识库更好，而非用完即弃
- TextGrad (Nature 2025) — 批评=梯度信号，自校准机制的学术基础
- 免疫系统类比 — 巡逻细胞(扫描) + 诊断细胞(分析) + 信号细胞(通知) + 记忆细胞(校准)
