# Knowledge Healer：知识库自愈巡检系统 v1.1

> 企业知识库的免疫系统——不是等人发现文档过期，而是主动发现、诊断、推动修复。
> 
> 设计灵感：Karpathy LLM Wiki（"Wiki是代码库，LLM是程序员"）+ 知识复利理论（每次交互让知识库更好）

---

## 核心理念

```
知识库 = 活的有机体，需要持续的新陈代谢才能存活

  健康态                      病态
  ├─ 内容新鲜                 ├─ 过期（Staleness）
  ├─ 内容一致                 ├─ 冲突（Contradiction）
  ├─ 引用完整                 ├─ 断链（Broken Links）
  ├─ 与实践同步               ├─ 漂移（Drift）
  ├─ 管线畅通                 ├─ 管线泄漏（Leakage）← v1.1新增
  ├─ 入口不堵塞               ├─ 捕获箱老化（Inbox Aging）← v1.1新增
  └─ 覆盖完整                 └─ 覆盖空洞（Coverage Gap）← v1.1新增

本skill = 免疫系统
  ├─ 巡逻细胞（定时扫描）→ 发现异常
  ├─ 诊断细胞（多Agent分析）→ 确认病因
  ├─ 信号细胞（通知推送）→ 调动修复
  └─ 记忆细胞（自校准）→ 越巡越准
```

**为什么需要**：

知识库最大的敌人不是"没人写"，而是三种隐性死亡：
1. **过期信息比没有信息更危险**——它给人虚假的确定感（新人按旧文档操作被坑）
2. **管线泄漏让复利归零**——内容散落在群聊/个人文档中绕过知识库，集体智慧无法积累
3. **入口堵塞让系统停摆**——捕获箱越堆越多无人处理，新信息永远进不了正式知识库

47%知识工作者每天找不到需要的信息，平均浪费1.8小时/天搜索。根因是"文档在但不可信"+"信息散落在体系之外"。

## 七维健康检测

### 1. 过期检测（Staleness）

**不是简单的"最后编辑>N天"**——不同类型文档有不同生命周期：

| 文档类型 | 默认有效期 | 判定逻辑 |
|----------|-----------|----------|
| 流程/SOP | 60天 | 最后编辑距今 > 有效期 AND 相关任务/群聊中有流程讨论 |
| 技术方案 | 90天 | 最后编辑距今 > 有效期 AND 代码仓库有相关变更 |
| 政策/制度 | 180天 | 最后编辑距今 > 有效期（政策类容忍度高） |
| 会议纪要 | ∞ | 永不过期（历史记录） |
| 入职指南 | 30天 | 最后编辑距今 > 有效期 AND 新人群有相关提问 |
| 产品文档 | 45天 | 最后编辑距今 > 有效期 AND 产品有版本更新 |

**加权信号**（多信号叠加才判定过期）：
- 编辑时间信号（基础权重0.3）
- 引用频率下降信号（权重0.2）
- 外部变化信号（群聊/任务/日历中讨论了相关内容但文档未更新，权重0.5）

**自校准机制**：每次巡检后记录"误报"（文档确认仍有效）和"漏报"（文档确实过期但未检出），动态调整有效期阈值和权重。

---

### 2. 冲突检测（Contradiction）

同一主题的多份文档描述不一致时，读者不知道"以哪个为准"。

**检测策略**：
```
Step 1: 按主题聚类（从标题/标签/目录位置推断主题归属）
Step 2: 同主题文档两两对比关键断言
Step 3: 判定冲突类型：
  - 版本冲突：同一流程的旧版/新版共存（新版未标注替代关系）
  - 视角冲突：不同部门对同一事物的描述不一致
  - 时效冲突：引用了不同时间点的数据/政策
Step 4: 评估冲突严重度：
  - P0：涉及合规/安全/财务的冲突 → 必须立即修复
  - P1：涉及核心流程的冲突 → 一周内修复
  - P2：非关键信息冲突 → 标记但不强制
```

**关键洞察**：冲突往往不是"两个都错"——通常一个是最新版但未传播，另一个是旧版但仍被引用。诊断重点是找出"权威源"。

---

### 3. 断链检测（Broken Links）

飞书文档/Wiki中的链接指向已删除、已移动或无权限的目标。

**检测范围**：
- 文档内链接 → 目标文档是否存在
- Wiki节点引用 → 目标节点是否仍在该空间
- 外部链接 → HTTP状态码检查（可选，默认关闭避免请求过多）
- 隐式引用 → 文档中提到"参见XX文档"但没有超链接（模糊匹配）

**修复建议生成**：
- 如果目标被移动 → 提供新地址
- 如果目标被删除 → 标记为"引用已失效"，建议删除或替换
- 如果是权限问题 → 标记为"需扩展权限"或"引用不当"

---

### 4. 漂移检测（Drift）— 本skill的核心创新

**定义**：文档内容与组织实际操作之间的偏差。文档没变，但现实变了。

**为什么这最难也最有价值**：
- 过期/冲突/断链都能从文档元数据推断
- 漂移需要**跨数据源交叉验证**——对比文档内容 vs 群聊讨论 vs 任务执行模式

**检测方法**：
```
Step 1: 提取文档核心断言（每份文档→3-10条关键声明）
        例：文档写"报销流程：填单→直属审批→财务审批"

Step 2: 在相关群聊中搜索反驳信号
        例：群里有人说"现在报销不用直属审批了，直接提交财务"

Step 3: 在任务/日历中搜索佐证
        例：最近的报销相关任务确实没有"直属审批"步骤

Step 4: 综合判定漂移概率
        - 单一群聊提及 → 低置信度（可能是个别情况）
        - 多人多次提及 → 中置信度（可能已变）
        - 任务/流程数据佐证 → 高置信度（确认已变）

Step 5: 生成漂移报告
        - 原文档断言
        - 反驳证据（来源+时间+原文）
        - 置信度评级
        - 建议：更新文档 / 确认是否已变 / 暂不处理
```

**约束**：
- chat_groups参数为空时，跳过漂移检测（改为quick模式）
- 漂移检测的搜索深度受群聊消息量限制，默认回溯30天
- 置信度<60%的漂移不进入最终报告（减少噪声）

### 5. 管线泄漏检测（Pipeline Leakage）— v1.1新增

**定义**：有价值的知识散落在非正式渠道（群聊、个人文档、会议录屏），绕过了知识库的正式录入流程。结果：集体智慧无法沉淀，知识复利归零。

**现实案例**：
- 群聊里反复回答同一个问题，但没人把答案写进wiki
- 某人的个人文档里有关键技术方案，但只有他自己知道
- 会议中做了重要决策，妙记录了但没人整理进知识库

**检测方法**：
```
Step 1: 在chat_groups中扫描"高价值信号"：
        - 被多人收藏/点赞的消息
        - 包含流程说明（"步骤1/2/3"、"首先...然后..."）的长消息
        - 回答了问题的消息（有人问→有人详细回答）
        - 包含文件/链接分享 + 解释的消息

Step 2: 检查这些高价值内容是否已存在于知识库中
        - 提取核心主题关键词
        - 在wiki中搜索匹配文档
        - 匹配度<30% → 判定为"泄漏"

Step 3: 评估泄漏严重���
        - 同一问题被问≥3次 + wiki无对应文档 → P0（反复浪费团队时间）
        - 高价值内容存在但未录入 → P1（知识只在个人脑中）
        - 一次性讨论，价值可能有限 → P2（标记观察）

Step 4: 生成"建议录入"清单
        - 内容摘要
        - 原始来源（群聊/文档链接）
        - 建议放入wiki的位置
        - 预估录入工作量（小/中/大）
```

**核心洞察**：知识库不长是因为"喂不进去"，而不是"没有内容"。内容一直在产生，只是散落在体系外。

### 6. 捕获箱老化检测（Inbox Aging）— v1.1新增

**定义**：待处理的内容（草稿、待审核文档、标记为inbox的条目）在入口堆积过久，导致信息流入正式知识库的管道堵塞。

**为什么重要**：
- 捕获箱是知识库的"入口"——如果入口堵了，新信息永远进不了体系
- 堆积会产生心理阻力——越堆越多→越不想处理→越堆越多（恶性循环）
- 过期的捕获内容如果最终被强制录入，可能引入已过时的信息

**检测方法**：
```
Step 1: 扫描所有"待处理"状态的内容
        - wiki中标记为"草稿"的页面
        - 文档中带有待办标��的条目
        - 知识库入口区（如特定文件夹）中的未处理项

Step 2: 计算滞留时间
        - 创建时间 vs 当前时间
        - >7天 → 轻度老化（提醒）
        - >14天 → 中度��化（建议尽快处理或放弃）
        - >30天 ��� 重度老化（内容可能已过期，需要重新评估是否还值得录入）

Step 3: 分类建议
        - 仍有时效性 → "建议尽��处理"
        - 已过时效 → "建议归档或删除，不再录入"
        - 属于长期参考资料 → "不受时效限制，降低优先级"

Step 4: 输出"入口健康度"
        - 入口流量：过去30天有多少新内容进入捕获箱
        - 处理速率：过去30天有多少内容被正式录入
        - 堆积量：当前未处理总量
        - 健康信号：处理速率 ≥ 入口���量 → 健康；< 50% → 堵塞
```

**自愈建议**：当检测到堵塞时，系统建议：
- 批量归档明显过期的条目（减少心理负担）
- 按价值排序——先处理最高价值的3条（而非按时间）
- 设置"30天未处理自动标记为expired"的规则

### 7. 覆盖空洞检��（Coverage Gap）— v1.1新增

**定义**��知识库的骨架存在（有目录/索引/空页面），但实际内容缺失。"框架陷阱"——结构设计得很漂亮，但里面是空的。

**为什么重要**：
- 空目录给人"这块已经覆盖了"的错觉——没人去填充因为以为"有人在管"
- 索引文件列出了条目但链接指向空页面——比没有索引更糟（期望落空）
- 频繁被搜索/访问但内容稀薄的页面 = 高ROI填充目标

**检测方法**：
```
Step 1: 扫描知识库目录结构
        - 识别所有"骨架节点"（有标题但内容<100字的页���）
        - 识别空目录（文件夹存在但内容为0或只有readme占位）
        - 识别索引中引用但不存在的页面

Step 2: 评估空洞的严重度（基于需求信号）
        - 该主题在群聊中被反复讨论 → P0（高需求但无文档覆盖）
        - 该主题有相关任务/项目在进行 → P1（应该有文档但没有）
        - 纯结构占位，暂无明确需求 → P2（可暂缓）
        
Step 3: 优先级排序标准
        "被问到的空洞" > "被引用的空洞" > "纯结构的空洞"
        
Step 4: 生成"建议填充"清单
        - 空洞位置
        - 需求证据（群聊提问/项目需要/高频搜索）
        - 建议内容来源（群聊讨论可直接整理？已有散落内容可聚合？）
        - 预估工作量
```

**与"管线泄漏"的协同**：
- 管线泄漏发现了"体系外存在的���容"
- 覆盖空洞发现了"体系内缺失的位置"
- 两者匹配 = 自动生成"从哪里取内容填到哪里"的修复方案

## 知识复利健康度指标（v1.1新增）

除了检测具体问题，系统还计算整体的"知识复利速率"：

```
复利速率 = 知识库净增长 / 组织信息产出总量

具体指标：
├─ 录入率：过去30天 新增wiki页面数 / 捕获箱入口数
├─ 引用率：wiki页面被引用次数 / 总页面数（活跃度）
├─ 更新率：过去30天被编辑的页面数 / 总页面数
├─ 泄漏率：群聊高价值内容数 / 其中已录入wiki数
└─ 综合复利指数：以上四率的加权平均
```

| 复利指数 | 状态 | 含义 |
|---------|------|------|
| >0.7 | 🟢 健康增长 | 知识在积累，系统活跃 |
| 0.4-0.7 | 🟡 增长放缓 | 有使用但维护不足 |
| 0.2-0.4 | 🟠 停滞 | 知���库开始"死亡"，需要干预 |
| <0.2 | 🔴 萎缩 | 只出不进，知识库名存实亡 |

## 执行流程

### Phase 0: 初始化

```
Step 1: 确认scope有效性（wiki/doc token是否可访问）
Step 2: 创建项目目录结构：
        {project_dir}/
        ├─ reports/          # 巡检报告
        ├─ baselines/        # 历史基线（用于对比）
        ├─ drift-evidence/   # 漂移证据归档
        ├─ rules/            # 自校准规则
        └─ state.json        # 巡检状态（支持中断恢复）
Step 3: 加载baseline（如有）→ 差分巡检（只检查变化部分）
Step 4: 加载custom_rules（如有）→ 覆盖默认阈值
```

### Phase 1: 文档采集

```
Step 1: 遍历scope内所有文档节点
        lark-cli wiki list-nodes --space {space_token} --depth {depth}
        记录：token / 标题 / 最后编辑时间 / 编辑者 / 父节点

Step 2: 按文档量决定策略：
        ≤ 50 文档 → 全量读取内容
        51-200 文档 → 读取元数据 + 采样20%内容
        > 200 文档 → 分批执行（每批50篇，batch模式）

Step 3: 构建文档图谱（adjacency map）：
        - 文档间引用关系
        - 目录层级关系
        - 标签/主题聚类
        
Step 4: 写入 state.json 标记"采集完成"（中断恢复点）
```

### Phase 2: 多Agent并行诊断

**根据mode选择诊断范围**：

| mode | 执行内容 |
|------|---------|
| full | 过期+冲突+断链+漂移（全部） |
| quick | 过期+断链（仅元数据级，不读内容） |
| drift-only | 仅漂移检测（已有基线时的增量检测） |
| scheduled | 同full，但静默执行+结果推送到指定群 |

**Agent分工**（full模式）：

```
主CC（调度者+综合者）
  │
  ├─ spawn Agent-Staleness
  │   职责：遍历文档列表，按类型匹配有效期，标记可疑过期文档
  │   输入：文档元数据列表 + 有效期规则 + 历史基线
  │   输出：{project_dir}/reports/staleness-findings.md
  │   约束：每篇文档判定必须附带判定理由（不是只输出结果）
  │
  ├─ spawn Agent-Contradiction
  │   职责：对同主题文档进行关键断言对比
  │   输入：文档内容（已按主题聚类）+ 冲突判定标准
  │   输出：{project_dir}/reports/contradiction-findings.md
  │   约束：只报告中高置信度冲突；低置信度的写入"待确认"区
  │
  ├─ spawn Agent-Links
  │   职责：检查所有文档内链接的有效性
  │   输入：文档内容中提取的链接列表
  │   输出：{project_dir}/reports/broken-links-findings.md
  │   约束：区分"确认断裂"vs"疑似断裂（权限问题）"
  │
  ├─ spawn Agent-Drift（仅当chat_groups非空）
  │   职责：提取文档核心断言 → 在群聊中搜索反驳信号
  │   输入：文档核心断言列表 + chat_groups列表 + 回溯天数
  │   输出：{project_dir}/drift-evidence/drift-findings.md
  │   约束：置信度<60%不进入主报告；必须附带原始证据引用
  │
  ├─ spawn Agent-Leakage（仅当chat_groups非空）— v1.1新增
  │   职责：在群聊中发现高价值但未被录入知识库的内容
  │   输入：chat_groups + 知识库索引（用于匹配已有内容）
  │   输出：{project_dir}/reports/leakage-findings.md
  │   约束：只报告被≥2人认可(收藏/点赞/回复确认)的内容
  │
  └─ 主CC直接执行（不需要spawn）：
      - Inbox Aging：扫描待处理区计算滞留时间（纯元数据操作，成本低）
      - Coverage Gap：对比目录结构 vs 实际内容量（纯结构分析）
```

**Spawn Prompt核心字段**（每个Agent必须包含）：
1. 研究范围（scope边界）
2. 输出文件绝对路径
3. 判定标准（什么算"发现"）
4. 证据要求（每个发现必须附带依据）
5. 字数硬限（findings文件严格不超过3000字/agent）
6. 安全边界（不修改任何文档，只读+报告）

### Phase 3: 综合诊断报告

```
Step 1: Read所有Agent产出文件（Glob验证存在且非空）

Step 2: 交叉验证（关键步骤）：
        - 过期文档中是否有漂移证据佐证？（有→提升优先级）
        - 冲突文档中是否有一方已过期？（有→明确谁是权威源）
        - 断链是否指向已删除的过期文档？（有→合并为同一修复项）

Step 3: 优先级排序（严重度 × 影响面）：
        - P0（必须修复）：合规/安全相关 OR 高流量文档过期 OR 有证据的漂移
        - P1（建议修复）：核心流程冲突 OR 多处断链 OR 中等过期
        - P2（标记观察）：低优先级过期 OR 单一低置信度信号

Step 4: 生成健康度评分（0-100）：
        score = 100 - (P0数×15 + P1数×5 + P2数×1)
        下限为0，分数越低越需要立即干预

Step 5: 写入综合报告 {project_dir}/reports/health-report.md
```

### Phase 4: 输出与通知

```
Step 1: 生成最终报告（Markdown格式，结构见下方模板）

Step 2: 如果notify=true：
        - 对每个P0/P1发现，@文档owner发送飞书消息
        - 消息格式："[知识库巡检] 您维护的《{文档标题}》被检测为{问题类型}，
          建议{修复动作}。详情：{报告链接}"
        - 批量发送，每人合并为一条（不单独骚扰）

Step 3: 如果baseline_path指定：
        - 对比本次 vs 上次：新增问题 / 已修复问题 / 恶化问题
        - 生成趋势图数据（JSON格式，可用于后续可视化）

Step 4: 保存本次结果为新baseline：
        {project_dir}/baselines/{timestamp}.json

Step 5: 如果mode=scheduled：
        - 将摘要推送到指定群聊
        - 格式："📋 知识库健康报告 {date}\n评分：{score}/100\n
          P0: {n}项 | P1: {n}项 | 较上次：↑/↓{n}项\n详情：{doc_link}"
```

## 自校准机制（Self-Calibration）

**灵感来源**：免疫系统的记忆B细胞——第一次遇到病原体反应慢，第二次遇到同类病原体立即精准响应。

### 校准循环

```
每次巡检完成后：

1. 收集反馈信号：
   - 用户标记"误报"（文档确实没问题）→ 放宽该类型的阈值
   - 用户标记"漏报"（有问题但没检出）→ 收紧该类型的阈值
   - P0修复速度（快修=真P0，长期不修=可能是P1）

2. 更新规则文件 {project_dir}/rules/calibration.yaml：
   staleness_thresholds:
     process_doc: {当前值} → {调整值}
     tech_spec: {当前值} → {调整值}
   drift_confidence_floor: {当前值} → {调整值}
   
3. 记录校准历史：
   - 哪些规则被调整 + 为什么 + 调整幅度
   - 连续3次同方向调整 → 标记为"稳定阈值"不再频繁调整
   
4. 校准上限保护：
   - 单次调整幅度 ≤ 20%（防止过拟合单次反馈）
   - 有效期阈值不低于14天（最低下限）
   - 置信度阈值不低于40%（最低下限）
```

### 反馈收集方式

```
方式1（主动）：巡检报告中附带反馈入口
  "本条发现是否准确？[准确] [误报] [需补充]"
  → 用户回复后更新calibration.yaml

方式2（被动）：追踪修复行为
  - P0报告后7天内文档被编辑 → 确认为真阳性
  - P0报告后30天未处理 → 可能是误报或优先级判定过高
  
方式3（显式）：用户可直接编辑rules/calibration.yaml
  - 添加白名单（某些文档不巡检）
  - 调整特定空间的阈值
  - 添加自定义规则
```

## 执行模式详解

### full（完整巡检）
适用：首次运行 / 季度全面体检
耗时：取决于文档量（50篇≈5分钟，200篇≈20分钟）
产出：完整报告 + 新baseline

### quick（快速巡检）
适用：日常/周度巡检
只做：过期检测（元数据） + 断链检测（链接校验）
跳过：冲突分析 + 漂移检测（需要读取内容和群聊，成本高）
耗时：50篇≈1分钟

### drift-only（漂移专项）
适用：已有基线，只想检测"现实是否偏离文档"
前提：必须提供chat_groups参数
耗时：取决于群聊消息量和文档断言数

### scheduled（定时执行）
适用：无人值守的定期巡检
行为：等同full，但静默执行 + 结果推送到群聊
配合：ScheduleWakeup 或 /loop 实现周期性执行
推荐频率：quick每周 / full每月 / drift-only每两周

## 安全与限制

1. **只读原则**：巡检过程绝不修改任何现有文档。只创建报告文档和发送通知。
2. **权限尊重**：无权限访问的文档标记为"无法检测"而非报错终止。
3. **频率限制**：scheduled模式最高频率为每天1次（避免API限流）。
4. **隐私保护**：漂移检测引用群聊消息时，脱敏处理个人信息，只保留与文档相关的事实性内容。
5. **噪声控制**：宁可漏报也不误报——误报会导致用户忽略所有警告（"狼来了"效应）。

## 设计哲学

### 为什么是"自愈"而不是"审计"

审计 = 找问题 + 报告 → 人来修。
自愈 = 找问题 + 诊断 + 推动修复 + 验证修复 + 记忆学习。

完整闭环：
```
检测 → 诊断 → 通知owner → 追踪修复 → 确认修复 → 更新baseline → 校准规则
 ↑                                                              │
 └──────────────── 下一轮巡检 ←─────────────────────────────────┘
```

### 为什么多Agent而不是单Agent

| 维度 | 单Agent | 多Agent |
|------|---------|---------|
| 判断独立性 | 一个Agent的bias影响全部结论 | 各Agent独立判断，交叉验证 |
| 专业度 | 通用prompt，各项均浅 | 每个Agent有专项prompt和判定标准 |
| 容错性 | 一个环节失败全链路停止 | 单Agent失败不影响其他维度 |
| 并行性 | 串行执行，耗时长 | 并行执行，总耗时≈最长单Agent |
| Context效率 | 上下文混杂所有检测维度 | 每个Agent只需持有本维度相关信息 |

### 为什么需要自校准

固定阈值的问题：
- 不同团队的文档更新节奏不同（研发团队快，行政部门慢）
- 不同业务阶段的"正常"不同（冲刺期文档更新快，维护期慢是正常的）
- 误报积累信任损耗——如果巡检报告50%是误报，用户会停止关注

自校准 = 让系统适应组织的实际节奏，而非强加外部标准。
